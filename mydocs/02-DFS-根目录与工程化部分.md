# 02 DFS-根目录与工程化部分

> 本卷聚焦非核心工程化能力（规范、构建、测试、发布）。

## 根目录（DFS 第一层）
- `.github/`：协作与自动化流程（Issue 模板、CI/CD、发布策略）
- `deploy/`：打包与发布脚本
- `docs/`：开发与运维文档
- `tests/`：测试入口
- 根级配置：`pyproject.toml`、`uv.lock`、`.gitignore`、`LICENSE`

## `.github/`
### 技术：GitHub Actions 工作流
- 技术是什么：云端自动化流水线。
- 为什么用：PR 自动测试，Tag 自动构建发布。
- 仓库里在哪里：`.github/workflows/build-release.yml`、`test-check.yml` 等。
- 对项目价值：让发布流程稳定、可追溯、可复现。

### 技术：Issue 模板与自动化处理
- 技术是什么：结构化问题收集与状态流转自动化。
- 为什么用：减少无效反馈，提高维护效率。
- 仓库里在哪里：`.github/ISSUE_TEMPLATE/*` 与 issue 自动化工作流。
- 对项目价值：降低沟通成本，提升问题定位速度。

### 技术：隐私政策文档
- 技术是什么：遥测与数据使用边界说明。
- 为什么用：明确用户知情权与数据处理范围。
- 仓库里在哪里：`.github/PRIVACY.md`。
- 对项目价值：增强用户信任，降低合规风险。

## `deploy/`
### 技术：PyInstaller 打包
- 技术是什么：将 Python 程序打包为 exe。
- 为什么用：非开发者可直接运行。
- 仓库里在哪里：`deploy/OneDragon-Installer.spec`、`deploy/OneDragon-Launcher.spec`。
- 对项目价值：显著降低安装门槛。

### 技术：批处理构建脚本
- 技术是什么：Windows `.bat` 自动化脚本。
- 为什么用：一键执行本地构建与资源整理。
- 仓库里在哪里：`deploy/build_full.bat`。
- 对项目价值：统一人工构建步骤，减少漏操作。

## `docs/`
### 技术：架构与模块文档体系
- 技术是什么：代码外的设计知识库。
- 为什么用：让新贡献者理解“为什么这样设计”。
- 仓库里在哪里：`docs/architecture/*`、`docs/develop/*`、`docs/运维指南/*`。
- 对项目价值：团队知识可沉淀、可传递。

## `tests/`
### 技术：pytest 测试
- 技术是什么：Python 自动化测试框架。
- 为什么用：避免改动引入回归错误。
- 仓库里在哪里：`tests/**/*`（CI 还会联动外部 `zzz-od-test`）。
- 对项目价值：提升重构与迭代安全性。

## 根级规范文件
### 技术：`pyproject.toml`
- 技术是什么：Python 项目统一元配置。
- 为什么用：集中管理依赖、lint、版本。
- 仓库里在哪里：仓库根目录。
- 对项目价值：环境和规范可复现。

### 技术：`uv.lock`
- 技术是什么：依赖锁文件。
- 为什么用：保证所有机器安装到同一依赖集合。
- 仓库里在哪里：仓库根目录。
- 对项目价值：减少“我这能跑你那不能跑”问题。

### 技术：`.gitignore`
- 技术是什么：Git 忽略规则。
- 为什么用：避免临时文件和构建产物污染仓库。
- 仓库里在哪里：仓库根目录。
- 对项目价值：保持版本历史整洁。

### 技术：`LICENSE`
- 技术是什么：开源许可声明。
- 为什么用：明确可用范围与责任边界。
- 仓库里在哪里：仓库根目录。
- 对项目价值：降低法律歧义。
